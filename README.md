## 1. Простая формулировка задачи

Сейчас в горно-металлургических R&D знания **«размазаны»** по отчётам,
презентациям, патентам и личным архивам. Из-за этого:

- теряются уже найденные решения (обессоливание воды, циркуляция католита,
  распределение Au/Ag/МПГ между штейном и шлаком),
- тратятся усилия на повторные обзоры,
- трудно искать связи между «режим → материал → процесс → результат»,
- не видно, кто в компании является экспертом по конкретной теме,
- появляются противоречивые выводы.

**Решение:** построить **граф знаний R&D** для горно-металлургической
отрасли, который:

1. **Импортирует** научные статьи (RU/EN), внутренние отчёты, протоколы
   экспериментов, патенты.
2. **Извлекает** NLP-пайплайном сущности (Material, Process, Equipment,
   Property, Experiment, Publication, Expert, Facility), связи (`uses_material`,
   `operates_at_condition`, `produces_output`, `described_in`, `validated_by`,
   `contradicts`) и **числовые ограничения** (мг/л, °C, т/сут и т.п.).
3. **Хранит** это в графовой БД (рекомендуется **Neo4j + Cypher**) с
   версионированием фактов и метаданными достоверности.
4. **Отвечает** на сложные многопараметрические запросы вида
   «материал + процесс + условия + география + диапазон дат» — простым
   исследователем на естественном языке, без знания графовой БД.
5. **Возвращает** структурированный ответ: сводка источников, консенсусные
   выводы, противоречия, пробелы в знаниях, рекомендации по смежным кейсам и
   экспертам.
6. **Поддерживает ролевую модель** (исследователь, аналитик, руководитель,
   админ, внешний партнёр), аудит и требования ИБ.
7. **Визуализирует** цепочки «материал → процесс → оборудование → результат»
   и подсвечивает пробелы / противоречия.

Целевые KPI:
- время ответа ≤ **3–5 сек** при обходе 3–4 уровней связей и до **1 млн** сущностей,
- корректное извлечение числовых ограничений и терминов (RU ↔ EN),
- интуитивный интерфейс без специальной подготовки.

## 2. План реализации

### 0. Подготовка (день 1)
- [ ] Зарегистрировать команду/проект на платформе хакатона.
- [ ] Скачать и разархивировать данные кейса (`disk.yandex.ru/d/npigiuw4Rbe9Pg`).
- [ ] Поднять репозиторий (Git), выбрать моно- или поли-репо, описать
      README, лицензию, базовый CI.
- [ ] Определить стек:
      **Neo4j 5.x** (Cypher, APOC, GDS) + **Python** (FastAPI) +
      **spaCy / DeepPavlov / ruBERT** для NLP +
      **Elasticsearch/Vespa** для полнотекста + **React/Streamlit** для UI +
      **OpenAI/Anthropic-compatible LLM** для NL→Cypher и синтеза ответов.

### 1. Онтология и модель графа (день 1)
- [ ] Зафиксировать лейблы: `Material | Process | Equipment | Property |
      Experiment | Publication | Expert | Facility | Condition`.
- [ ] Зафиксировать типы связей: `uses_material, operates_at_condition,
      produces_output, described_in, validated_by, contradicts, authored_by,
      affiliated_with, measured_by`.
- [ ] Свойства узлов минимум: `id, name_i18n (ru/en), aliases[], tags[]`.
- [ ] Свойства связей минимум: `confidence (0..1), source_date, source_ref,
      numeric_value, unit, valid_from, valid_to`.

### 2. Импорт данных (день 1–2)
- [ ] Коннекторы: внутренние отчёты (PDF/DOCX), статьи (PDF), патенты (XML),
      таблица экспериментов (CSV/Excel), справочник экспертов (CSV).
- [ ] Чеканка и конвертация в **Markdown/JSON-LD**.
- [ ] Нормализация единиц измерения; сопоставление синонимов RU ↔ EN
      (словарь + Tf-Idf/embeddings).
- [ ] Версионирование источников.

### 3. NLP-пайплайн (день 2–3)
- [ ] Извлечение сущностей (NER): ruBERT / spaCy ru/en; gazetteer усиление.
- [ ] Извлечение связей (Relation Extraction) — правила + модель.
- [ ] Извлечение числовых ограничений с единицами (регексы + NER для ед.).
- [ ] Дедупликация и слияние с уже существующими узлами (MERGE в Cypher).
- [ ] Контроль качества: ручная разметка выборки, метрики precision/recall.

### 4. Загрузка в Neo4j (день 3)
- [ ] Скрипты-импортёры (`MERGE`-based), идемпотентные.
- [ ] Индексы: `Expert(name)`, `Material(name)`, `Process(name)`,
      `Publication(doi)`, `Experiment(id)`.
- [ ] Constraints (`UNIQUE`) на естественные ключи.
- [ ] APOC-процедуры для батч-импорта и онтологических проверок.
- [ ] Тестовые Cypher-запросы по эталонным 5–10 кейсам из ТЗ.

### 5. Поиск / запросы (день 3–4)
- [ ] Полнотекстовый индекс (Elasticsearch/Vespa) для семантического поиска.
- [ ] **NL→Cypher** через LLM с few-shot примерами + валидацией синтаксиса.
- [ ] Шаблонные конструкторы запросов под типовые кейсы из ТЗ (готовые
      Cypher-скелеты для пользователя).
- [ ] Фильтры: материал, процесс, география, год, диапазон чисел,
      уровень достоверности.
- [ ] Сравнительные запросы: RU vs зарубежная практика, вариант A vs B.

### 6. Аналитика и синтез ответа (день 4)
- [ ] Группировка источников по методу/году/географии.
- [ ] Выделение консенсуса и зон противоречий (`contradicts`).
- [ ] Детекция пробелов: «нет экспериментов для комбинации X».
- [ ] Рекомендации: похожие кейсы + список экспертов.
- [ ] Оценка уверенности (по `confidence`, количеству источников).

### 7. Визуализация (день 4)
- [ ] Граф-просмотрщик цепочек «материал → процесс → оборудование →
      результат» (vis.js / React + d3 / Neo4j Browser).
- [ ] Подсветка противоречий и пробелов.
- [ ] Дашборд для руководителей: покрытие знаний по доменам, активность,
      зоны риска.

### 8. UI / UX (день 4–5)
- [ ] Поле ввода запроса на естественном языке + кнопка «уточнить фильтры».
- [ ] Карточка ответа: текст, граф-цепь, список источников, конфликты,
      рекомендации, эксперты.
- [ ] Режим «литературный обзор по клику».
- [ ] Экспорт: PDF / Markdown / JSON-LD.

### 9. Доступ / безопасность (день 5)

- [ ] Аудит запросов, просмотров, экспорта.
