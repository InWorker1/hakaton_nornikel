# Онтология графа знаний R&D (кейс «Научный клубок»)

> Цель — единый источник правды для лейблов узлов, типов связей и свойств,
> используемых во всех Cypher-скриптах, NLP-пайплайне и API.
>
> Связанные файлы:
> - `ontology/schema.cypher` — DDL: constraints + indexes (идемпотентные).
> - `ontology/ontology.py` — Python-модели для типизированной работы в коде.

Согласовано с разделом 1 (`Онтология и модель графа`) и разделом 2 (`Импорт
данных`) основного `README.md` проекта.

---

## 1. Узлы (лейблы)

| Лейбл         | Назначение (RU / EN)                              | Пример                                   |
|---------------|---------------------------------------------------|------------------------------------------|
| `Material`    | Материал / реагент / сырьё / конечный продукт     | «сульфидный никелевый концентрат», «Cu»  |
| `Process`     | Технологический процесс или операция              | «электролитическое рафинирование»        |
| `Equipment`   | Конкретный аппарат / установка / модуль           | «электролизёр серии Х-300»               |
| `Property`    | Измеряемое свойство (физическое/химическое)       | «плотность», «выход по току»             |
| `Experiment`  | Конкретный эксперимент / опыт / серия испытаний   | «опыт №42, серия 2024-Q2»                |
| `Publication` | Научная статья, отчёт, патент, протокол           | doi:`10.1234/abc`, патент RU 2718281      |
| `Expert`      | Сотрудник / автор / рецензент / внешний эксперт   | ORCID: `0000-0002-1825-0097`              |
| `Facility`    | Цех / лаборатория / завод / площадка              | «Норильская обогатительная фабрика»      |
| `Condition`   | Условия процесса (T°, P, pH, состав, расход и т.п.)| «120 °C», «pH 9.5», «расход 5 м³/ч»      |

> `Condition` — это узел-помощник для связей с числовыми ограничениями. В
> большинстве случаев достаточно хранить `numeric_value`/`unit` прямо на
> реляции (`operates_at_condition`), но если условие повторно используется
> в нескольких экспериментах — выносим в `Condition`.

### 1.1 Свойства узлов (общие + специальные)

Общие свойства **для каждого** лейбла:

| Свойство          | Тип            | Описание                                          | Инвариант                       |
|-------------------|----------------|---------------------------------------------------|----------------------------------|
| `id`              | STRING (UUID)  | Глобальный стабильный идентификатор                | `REQUIRE n.id IS UNIQUE`         |
| `slug`            | STRING         | URL-безопасное имя                                | уникален в пределах лейбла       |
| `name_i18n`       | MAP (Python)   | `{"ru": "...", "en": "..."}`                       | хотя бы один ключ непустой       |
| `name_i18n_ru`    | STRING (Cypher)| Плоская проекция `name_i18n.ru` для индексов       | создаётся индексом               |
| `name_i18n_en`    | STRING (Cypher)| Плоская проекция `name_i18n.en` для индексов       | создаётся индексом               |
| `aliases`         | LIST<STRING>   | Синонимы/аббревиатуры (RU + EN)                    | может быть пустой                |
| `tags`            | LIST<STRING>   | Теги домена/проекта                               | может быть пустой                |
| `created_at`      | DATETIME (UTC) | Когда узел создан                                  | ставится автоматически           |
| `updated_at`      | DATETIME (UTC) | Когда узел последний раз менялся                   | ставится автоматически           |

> В **Neo4j** узел физически хранит **два скалярных поля** `name_i18n_ru` и
> `name_i18n_en` — под них построены btree-индексы в `schema.cypher`.
> Map-форма `name_i18n` существует только в Python-модели (`ontology.py`) и
> в обменных JSON-сообщениях API. Сериализация: `node_to_props_dict()`
> раскрывает map → два скаляра.

Специальные свойства **по лейблам**:

| Лейбл        | Спец-свойства                                                |
|--------------|---------------------------------------------------------------|
| `Material`   | `formula: STRING`, `cas_number: STRING`, `family: STRING`     |
| `Process`    | `kind: STRING`  (обогащение/пиро/гидро/электро/проч.)         |
| `Equipment`  | `model: STRING`, `vendor: STRING`, `capacity: STRING`         |
| `Property`   | `kind: STRING` (`scalar`/`vector`/`distribution`), `unit: STRING` |
| `Experiment` | `started_at: DATE`, `ended_at: DATE`, `batch_id: STRING`, `sample_count: INTEGER` |
| `Publication`| `doi: STRING`, `year: INTEGER`, `venue: STRING`, `lang: STRING`, `url: STRING` |
| `Expert`     | `orcid: STRING`, `email: STRING`, `position: STRING`, `current: BOOLEAN` |
| `Facility`   | `country: STRING`, `city: STRING`, `kind: STRING`             |
| `Condition`  | `value: FLOAT`, `unit: STRING`, `kind: STRING` (`temperature`/`pressure`/`ph`/...) |

Все строковые значения хранятся в **utf-8**, регистр исходный (поиск
case-insensitive делается в приложении или через full-text index).

## 2. Связи (типы)

Во всех связях действует **system property bag**:

| Свойство         | Тип           | Описание                                         | Инвариант                              |
|------------------|---------------|--------------------------------------------------|----------------------------------------|
| `confidence`     | FLOAT (0..1)  | Уверенность извлечения NLP                       | `0 <= confidence <= 1`                 |
| `source_date`    | DATE          | Дата факта в источнике                           | не пусто                              |
| `source_ref`     | STRING        | URI/citation: `doi:…`, `report:NOR-2024-123`, …   | не пусто                              |
| `source_excerpt` | STRING        | Дословная цитата из источника                    | не пусто                              |
| `numeric_value`  | FLOAT         | Числовое значение                                | если задано, то `unit` тоже задан     |
| `unit`           | STRING        | Единица измерения (UCUM/ISO)                     | согласовано с `numeric_value`         |
| `valid_from`     | DATE          | Когда факт стал валиден                          | если не задан — факт «вечный»          |
| `valid_to`       | DATE          | Когда факт перестал быть валиден                 | если не задан — факт актуален          |
| `note`           | STRING        | Свободная пометка                                | —                                     |

### 2.1 Связи и их семантика

| Тип связи              | From → To                              | Семантика                                              |
|------------------------|----------------------------------------|--------------------------------------------------------|
| `uses_material`        | Process → Material                     | Процесс использует данный материал/реагент.            |
| `operates_at_condition`| Process → Condition                    | Процесс идёт при таком условии (T, P, pH, расход).     |
| `operates_at_condition`| Experiment → Condition                 | Конкретный эксперимент проведён при таком условии.    |
| `produces_output`      | Process → Material                     | Процесс производит данный продукт.                     |
| `produces_output`      | Process → Property                     | Процесс даёт такое измеряемое свойство на выходе.      |
| `measured_by`          | Property → Experiment                  | Значение свойства измерено в этом эксперименте.        |
| `measured_by`          | Property → Publication                 | Значение свойства опубликовано в этой работе.         |
| `conducted_at`         | Experiment → Facility                  | Эксперимент проведён на этой площадке.                 |
| `conducted_at`         | Experiment → Equipment                 | Эксперимент проведён на этом оборудовании.             |
| `described_in`         | любой → Publication                    | Сущность упомянута/описана в этой публикации.          |
| `authored_by`          | Publication → Expert                   | Авторство публикации.                                  |
| `affiliated_with`      | Expert → Facility                      | Аффилиация эксперта с площадкой/организацией.          |
| `validated_by`         | любой → Experiment                     | Факт/связь подтверждена этим экспериментом.            |
| `contradicts`          | любой ⇄ любой                          | Конфликтующие утверждения (направление двустороннее).  |
| `related_to`           | любой ⇄ любой                          | Резервная «мягкая» связь (для нечётких упоминаний).    |

> Двусторонние типы (`contradicts`, `related_to`) хранятся в **одном
> направлении** (например, `(A)-[:contradicts]->(B)`); обратное направление
> выражается либо отдельной реляцией, либо учитывается в запросах
> `MATCH (a)-[:contradicts]-(b)`.

## 3. Естественные ключи (для MERGE)

Для импорта важна идемпотентность. Используем:

| Лейбл         | Естественный ключ (MERGE)                             |
|---------------|-------------------------------------------------------|
| `Material`    | `id` (UUID) _или_ `(family, formula, cas_number)`     |
| `Process`     | `id` _или_ `(slug)` при наличии                       |
| `Equipment`   | `id` _или_ `(model, vendor)`                          |
| `Property`    | `id` _или_ `slug`                                    |
| `Experiment`  | `id` _или_ `(batch_id, started_at)`; `slug` обязателен как у всех узлов |
| `Publication` | `id` _или_ `doi` _или_ `(title, year)`                |
| `Expert`      | `id` _или_ `orcid` _или_ `(full_name, current)`       |
| `Facility`    | `id` _или_ `(name, country, city)`                    |
| `Condition`   | `id` _или_ `(kind, value, unit)`                      |

В импортёрах (п.4 плана) — **MERGE по `id`**, а узлы создаются с
предварительным определением `id` в NLP-слое.

## 4. Инварианты (валидация)

1. `id` каждой сущности — **уникален** в пределах своего лейбла
   (enforced unique constraint).
2. Если `numeric_value IS NOT NULL`, то и `unit IS NOT NULL`
   (проверяется в Python и/или APOC; см. `validate.cypher`).
3. `confidence` ∈ [0, 1] (проверяется в Python — у Cypher нет
   CHECK constraints в стиле SQL).
4. `valid_from <= valid_to`, если оба заданы (проверяется в импортёре).
5. `name_i18n` содержит хотя бы один непустой ключ (`ru` или `en`).
6. `orcid` и `doi` при наличии — формат валиден (regex в Python).
7. `contradicts` фиксируется всегда **вместе с пометкой `(сторона 1 vs
   сторона 2)` в `note`** (для аудита).

## 5. Эволюция схемы

- Новые лейблы добавляются **только** через этот файл + соответствующий
  блок в `schema.cypher` (см. шаблон в конце `schema.cypher`).
- Связь, не описанная здесь, **не должна** появляться в данных.
- При изменении свойств узлов — миграция в Neo4j делается через
  `MATCH … SET n.new_field = old_old_field` под управлением скрипта в
  `migrations/` (заведём в п.11).

## 6. Конвенция MERGE / импорта

- Импортёр делает `MERGE (n:Label {id: $id})` по `id` — **единственный**
  стабильный ключ, гарантированный unique-constraint.
- Альтернативные естественные ключи из таблицы в §3 используются **только**
  для предварительного маппинга «сырых записей → id»: если запись приходит
  без `id` (например, DOI для статьи), импортёр через `MATCH`-кандидат
  резолвит существующий `id` или создаёт новый.
- `slug` генерируется из `name_i18n` нормализатором (транслит RU→EN,
  только `[a-z0-9-]`), уникальность — constraint. При коллизии —
  к slug добавляется суффикс `-2`, `-3`, …
- Тип связи (например, `uses_material`) **не** хранится как свойство.
  В Cypher тип пишется в сам паттерн (`-[:uses_material]->`), в Python
  хранится в `BaseRelationship.type` (`RelationshipType` enum).
